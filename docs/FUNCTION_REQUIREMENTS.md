# 待办事项 App 功能需求文档

版本：1.0  
日期：2026-07-01  
项目代号：BEGO  
来源文档：`docs/SRS.md`  
适用范围：后端 MVP 功能开发、接口联调、测试验收

## 1. 文档目的

本文档从 SRS 中独立整理 BEGO 待办事项 App 的功能需求，面向后端开发和测试验收使用。本文档只描述业务功能、业务规则、接口能力和验收标准，不限定具体代码包结构。

后端代码结构可按常规 Spring Boot 项目分层组织，例如 Controller、DTO、VO、Entity、Service、Mapper/Repository。

## 2. 功能优先级

| 优先级 | 含义 |
| --- | --- |
| P0 | MVP 必须交付，缺失则主流程不可用。 |
| P1 | MVP 建议交付或测试版需要支持，可在 P0 稳定后完善。 |
| P2 | 后续阶段交付，不阻塞 MVP。 |

## 3. MVP 功能范围

MVP 必须包含：

- 用户注册、登录、退出。
- Access Token + Refresh Token。
- 当前用户信息查询。
- 用户注销。
- 待办创建、列表、详情、编辑、完成、删除。
- 离线创建待办后的幂等同步。
- 标签创建、列表、编辑、删除、排序。
- 待办关联多个标签。
- 关键词搜索和基础组合筛选。
- 提醒时间字段保存。
- Flyway 初始化数据库结构。

MVP 不包含：

- 图片上传、展示、删除。
- 图片缩略图。
- 对象存储或本地文件存储方案。
- 服务端推送通知。
- 团队协作。
- 子任务。
- 循环任务。

## 4. 用户账户功能

### 4.1 用户注册（P0）

用户应能使用邮箱、密码、昵称注册账号。

输入：

| 字段 | 规则 |
| --- | --- |
| email | 必填，邮箱格式，最大 255 个字符。 |
| password | 必填，至少 8 位。 |
| displayName | 必填，1-80 个字符。 |

业务规则：

- 邮箱归一化后必须唯一，归一化规则为 trim 后转小写。
- 密码必须哈希存储，不能明文保存。
- 注册成功后应自动签发 Access Token 和 Refresh Token。
- 新用户状态默认为 `ACTIVE`。

验收标准：

- 使用新邮箱注册成功。
- 重复邮箱注册返回明确错误。
- 注册成功响应包含当前用户信息、Access Token、Refresh Token。

### 4.2 用户登录（P0）

用户应能使用邮箱和密码登录。

输入：

| 字段 | 规则 |
| --- | --- |
| email | 必填 |
| password | 必填 |
| deviceName | 可选 |

业务规则：

- 登录时使用归一化邮箱查找用户。
- 用户不存在、密码错误时返回统一认证失败提示。
- 用户状态不是 `ACTIVE` 时禁止登录。
- 登录成功后更新最近登录时间。
- 登录成功后签发 Access Token 和 Refresh Token。

验收标准：

- 正确邮箱密码可登录。
- 错误密码不可登录。
- 已注销或禁用用户不可登录。

### 4.3 刷新令牌（P0）

系统应支持使用 Refresh Token 刷新 Access Token。

业务规则：

- 服务端只保存令牌哈希，不保存明文令牌。
- Refresh Token 必须存在、未撤销、未过期。
- 刷新成功后建议轮换 Refresh Token。
- 旧 Refresh Token 轮换后应失效。

验收标准：

- Access Token 过期后可使用有效 Refresh Token 获取新令牌。
- 无效、过期、已撤销的 Refresh Token 返回认证失败。

### 4.4 用户退出（P1）

用户应能退出当前设备登录。

业务规则：

- 退出时撤销当前设备相关 Access Token 和 Refresh Token。
- 客户端清理本地令牌。

验收标准：

- 退出后原 Access Token 不可继续访问受保护接口。
- 退出后需要重新登录。

### 4.5 当前用户信息（P0）

登录用户应能查询自己的基础信息。

响应字段：

| 字段 | 说明 |
| --- | --- |
| id | 用户 ID |
| email | 邮箱 |
| displayName | 昵称 |
| avatarUrl | 头像 URL，可为空 |

验收标准：

- 未登录访问返回 401。
- 登录后返回当前用户信息。

### 4.6 用户注销（P1）

用户应能注销自己的账号。

业务规则：

- 注销前需要二次确认，建议要求重新输入密码。
- 注销后用户状态改为 `DELETED`。
- 注销后用户 PII 应脱敏，包括邮箱、昵称、头像。
- 注销后软删除用户的待办和标签。
- 注销后撤销该用户全部 Access Token 和 Refresh Token。

验收标准：

- 注销后不能再使用原账号登录。
- 注销后原令牌全部失效。
- 注销后该用户待办和标签在普通查询中不可见。

## 5. 待办事项功能

### 5.1 创建待办（P0）

用户应能创建待办事项。

输入：

| 字段 | 规则 |
| --- | --- |
| title | 必填，1-120 个字符 |
| description | 可选，最多 5000 个字符 |
| priority | 可选，`LOW`、`MEDIUM`、`HIGH`，默认 `MEDIUM` |
| dueAt | 可选，截止时间 |
| reminderAt | 可选，提醒时间 |
| tagIds | 可选，标签 ID 列表 |
| clientTempId | 可选，离线创建临时 ID |

业务规则：

- 待办默认状态为 `TODO`。
- 关联的标签必须属于当前用户且未删除。
- `clientTempId` 非空时用于离线创建幂等同步。
- 创建成功后 `syncStatus` 为 `SYNCED`。

验收标准：

- 标题合法时创建成功。
- 标题为空或超长时返回校验错误。
- 关联其他用户标签时创建失败。

### 5.2 离线创建同步（P1）

前端离线创建待办后，网络恢复时应能同步到服务端。

业务规则：

- 前端离线创建时生成 `clientTempId`。
- 后端按 `(userId, clientTempId)` 做幂等判断。
- 如果已存在相同 `clientTempId` 的待办，直接返回已有待办。
- 如果不存在，则创建新待办。
- 同一个离线待办重复同步不得产生重复数据。

验收标准：

- 同一 `clientTempId` 重试 2 次只生成 1 条服务端待办。
- 同步成功后返回服务端 ID。
- 同步成功后 `syncStatus` 为 `SYNCED`。

### 5.3 待办列表（P0）

用户应能查看自己的待办列表。

查询条件：

| 参数 | 说明 |
| --- | --- |
| status | `TODO`、`DONE` |
| priority | `LOW`、`MEDIUM`、`HIGH` |
| tagId | 标签 ID |
| keyword | 标题和描述关键词 |
| dueFrom | 截止时间开始 |
| dueTo | 截止时间结束 |
| sort | 排序字段 |
| page | 页码 |
| size | 每页数量 |

业务规则：

- 只能查询当前用户数据。
- 默认过滤软删除数据。
- 筛选条件可以组合使用。
- 默认排序为未完成优先，其次截止时间升序，再更新时间降序。
- 分页大小应有上限，建议最大 100。

验收标准：

- 用户只能看到自己的待办。
- 标签、状态、优先级、关键词可组合筛选。
- 删除后的待办不出现在列表中。

### 5.4 待办详情（P0）

用户应能查看单个待办详情。

详情包含：

- 标题。
- 描述。
- 状态。
- 优先级。
- 截止时间。
- 提醒时间。
- 完成时间。
- 标签列表。
- 创建时间。
- 更新时间。
- 同步状态。

业务规则：

- 只能查看自己的未删除待办。
- 不存在、已删除、不属于当前用户的待办返回 404。

### 5.5 编辑待办（P0）

用户应能编辑自己的待办。

可编辑字段：

- 标题。
- 描述。
- 优先级。
- 截止时间。
- 提醒时间。
- 标签列表。

业务规则：

- 编辑时必须校验待办归属。
- 标签列表整体替换。
- 新标签列表中的标签必须属于当前用户。
- 修改后更新时间自动更新。

验收标准：

- 修改标题、描述、标签后详情正确展示。
- 使用其他用户标签时更新失败。

### 5.6 完成和取消完成（P0）

用户应能将待办标记为完成，也能取消完成。

业务规则：

- 标记为 `DONE` 时写入完成时间。
- 改回 `TODO` 时清空完成时间。
- 已删除待办不可更新状态。

验收标准：

- 完成后列表展示为已完成。
- 取消完成后状态恢复为未完成。

### 5.7 删除待办（P0）

用户应能删除自己的待办。

业务规则：

- 删除采用软删除。
- 删除后普通列表和详情不可见。
- 删除待办不删除标签本身。
- P2 附件功能引入后，附件也应随待办隐藏。

验收标准：

- 删除后列表不再展示。
- 删除后访问详情返回 404。

## 6. 标签功能

### 6.1 创建标签（P0）

用户应能创建自定义标签。

输入：

| 字段 | 规则 |
| --- | --- |
| name | 必填，1-30 个字符 |
| color | 可选，十六进制颜色，默认 `#2F80ED` |
| sortOrder | 可选，默认 0 |

业务规则：

- 同一用户下未删除标签名称不可重复。
- 标签名称归一化后做唯一性判断。
- 标签颜色必须是合法十六进制颜色。

验收标准：

- 创建标签成功后列表可见。
- 同一用户创建重复名称标签返回冲突错误。
- 删除后的同名标签允许重新创建。

### 6.2 标签列表（P0）

用户应能查看自己的标签列表。

响应包含：

- 标签 ID。
- 名称。
- 颜色。
- 排序值。
- 未完成待办数量。

业务规则：

- 只返回当前用户未删除标签。
- 默认按 `sortOrder` 升序，再按创建时间升序。
- 未完成待办数量只统计未删除且状态为 `TODO` 的待办。

### 6.3 编辑标签（P0）

用户应能编辑标签名称、颜色、排序值。

业务规则：

- 只能编辑自己的未删除标签。
- 修改名称后仍需保证同一用户下唯一。
- 修改标签不会影响待办本身。

验收标准：

- 修改后待办列表中的标签展示同步更新。
- 重复名称返回冲突错误。

### 6.4 删除标签（P0）

用户应能删除标签。

业务规则：

- 删除采用软删除。
- 删除标签不删除待办。
- 删除后待办不再展示该标签。
- 删除时设置软删除唯一约束辅助字段，允许后续重新创建同名标签。

验收标准：

- 删除后标签列表不可见。
- 删除后相关待办仍存在。

### 6.5 标签排序（P1）

用户应能调整标签展示顺序。

业务规则：

- 批量更新多个标签的 `sortOrder`。
- 所有标签必须属于当前用户。
- 排序更新应在事务内完成。

验收标准：

- 排序后标签列表按新顺序展示。

## 7. 提醒功能

### 7.1 提醒时间字段（P1）

用户应能为待办设置提醒时间。

业务规则：

- 提醒时间可独立于截止时间存在。
- 如果同时设置截止时间，建议提醒时间不晚于截止时间。
- 后端 MVP 只保存提醒时间，不负责服务端推送。

验收标准：

- 创建和编辑待办时可保存提醒时间。
- 查询详情时返回提醒时间。

### 7.2 本地通知配合（P1，测试版）

本地通知由 Flutter 客户端实现，后端仅提供提醒时间字段。

业务规则：

- 后端不直接触发本地通知。
- 已完成或已删除待办应由客户端取消本地通知。

## 8. 搜索和筛选功能

### 8.1 关键词搜索（P1）

用户应能按关键词搜索待办。

业务规则：

- 搜索范围为标题和描述。
- 只搜索当前用户未删除待办。
- MVP 可使用模糊匹配。

验收标准：

- 输入标题关键词可搜索到对应待办。
- 输入其他用户待办关键词不会返回结果。

### 8.2 组合筛选（P1）

用户应能组合状态、优先级、标签、截止时间范围筛选待办。

业务规则：

- 筛选条件可以组合。
- 标签筛选只接受当前用户标签。
- 筛选结果分页返回。

## 9. 图片附件功能（P2）

图片附件功能不纳入 MVP。

P2 预留功能：

- 上传图片。
- 查看图片缩略图。
- 查看原图。
- 删除图片。
- 图片存储方案。
- 图片访问权限校验。

MVP 约束：

- 不实现附件接口。
- 不创建附件表。
- 待办创建和编辑不依赖图片。

## 10. 权限和数据隔离

所有受保护接口必须满足：

- 当前用户只能访问自己的用户信息。
- 当前用户只能访问自己的待办。
- 当前用户只能访问自己的标签。
- 当前用户只能使用自己的标签关联待办。
- 软删除数据默认不可见。

越权访问处理：

- 查询不存在、不属于当前用户、已删除资源时，统一返回 404。
- 未认证返回 401。
- 认证成功但状态不允许操作时返回 403。

## 11. 错误场景

| 场景 | 建议错误码 | HTTP |
| --- | --- | --- |
| 参数校验失败 | `VALIDATION_ERROR` | 400 |
| 未登录 | `UNAUTHORIZED` | 401 |
| Access Token 过期 | `TOKEN_EXPIRED` | 401 |
| Refresh Token 无效 | `REFRESH_TOKEN_INVALID` | 401 |
| 用户被禁用 | `ACCOUNT_DISABLED` | 403 |
| 用户已注销 | `ACCOUNT_DELETED` | 403 |
| 资源不存在 | `RESOURCE_NOT_FOUND` | 404 |
| 邮箱已存在 | `EMAIL_ALREADY_EXISTS` | 409 |
| 标签名称重复 | `TAG_NAME_ALREADY_EXISTS` | 409 |
| 客户端临时 ID 冲突 | `CLIENT_TEMP_ID_CONFLICT` | 409 |
| 服务端异常 | `INTERNAL_ERROR` | 500 |

## 12. 验收主流程

### 12.1 在线主流程

1. 用户注册。
2. 用户登录。
3. 创建标签。
4. 创建待办并关联标签。
5. 查询待办列表。
6. 按标签筛选待办。
7. 编辑待办。
8. 标记待办完成。
9. 删除待办。
10. 退出登录。

### 12.2 离线创建同步流程

1. 客户端离线创建待办并生成 `clientTempId`。
2. 网络恢复后调用创建待办接口。
3. 服务端根据 `clientTempId` 幂等创建。
4. 客户端收到服务端 ID。
5. 重复提交同一 `clientTempId` 不产生重复待办。

### 12.3 注销流程

1. 用户登录。
2. 用户发起注销并二次确认。
3. 服务端脱敏用户信息。
4. 服务端软删除待办和标签。
5. 服务端撤销全部令牌。
6. 用户再次访问受保护接口返回 401。

## 13. 后端开发备注

- 本文档不规定代码包结构。
- 后端实现可按常规 Controller、DTO、VO、Entity、Service、Mapper 分层。
- 业务规则应主要落在 Service 层。
- Controller 负责请求参数、认证用户和响应转换。
- Mapper 负责 MyBatis SQL。
- Entity 对应数据库表。
- VO 用于响应给前端。
