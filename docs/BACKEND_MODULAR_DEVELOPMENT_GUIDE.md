# BEGO 模块化后端开发阶段指导书

版本：1.0  
日期：2026-07-01  
适用范围：BEGO 待办事项 App 后端 MVP 开发、接口联调、测试验收  
依据文档：`docs/SRS.md`、`docs/FUNCTION_REQUIREMENTS.md`、`docs/DATABASE_DESIGN.md`

## 1. 指导目标

本文档用于指导 BEGO 后端按模块、按阶段完成 MVP 开发。后端应围绕“用户账户 + 认证令牌 + 待办事项 + 标签 + 离线创建幂等同步 + Flyway 数据库迁移”构建稳定闭环。

开发时以 P0 功能为第一优先级，P1 功能在 P0 主流程稳定后补齐，P2 图片附件、服务端推送、子任务、循环任务和团队协作暂不进入 MVP 实现范围。

## 2. 技术与分层约定

### 2.1 技术栈

- Java 17。
- Spring Boot。
- Spring Security+JWT。
- MyBatis。
- MySQL 8.x。
- Flyway。
- Maven。

### 2.2 推荐包结构

后端主包建议使用 `com.bego.backend`，按业务模块与公共能力拆分：

```text
com.bego.backend
├── common
│   ├── error
│   ├── response
│   ├── security
│   ├── time
│   └── validation
├── user
├── auth
├── todo
├── tag
└── sync
```

每个业务模块内部建议保持一致分层：

```text
module
├── controller
├── dto
├── vo
├── entity
├── mapper
└── service
```

### 2.3 分层职责

| 层级 | 职责 |
| --- | --- |
| Controller | 接收请求、参数校验、获取当前用户、组装响应。 |
| DTO | 请求入参对象，承载校验注解。 |
| VO | 响应对象，避免直接暴露 Entity。 |
| Service | 业务规则、权限校验、事务边界、幂等处理。 |
| Mapper | MyBatis 数据访问与 SQL 映射。 |
| Entity | 数据库表结构映射。 |
| common | 统一错误响应、认证上下文、时间处理、工具能力。 |

业务规则必须主要落在 Service 层，Mapper 不承载业务判断，Controller 不直接拼接业务流程。

## 3. 全局开发规则

### 3.1 API 约定

- API 基础路径：`/api/v1`。
- 受保护接口使用 `Authorization: Bearer <token>`。
- 时间字段使用 ISO 8601 格式传输，服务端按 UTC 写入数据库。
- 分页响应统一：

```json
{
  "items": [],
  "page": 1,
  "size": 20,
  "total": 100
}
```

- 错误响应统一：

```json
{
  "code": "VALIDATION_ERROR",
  "message": "请求参数不合法",
  "details": [
    {
      "field": "title",
      "message": "标题不能为空"
    }
  ]
}
```

### 3.2 错误码约定

| 场景 | 错误码 | HTTP |
| --- | --- | --- |
| 参数校验失败 | `VALIDATION_ERROR` | 400 |
| 未登录 | `UNAUTHORIZED` | 401 |
| Access Token 过期 | `TOKEN_EXPIRED` | 401 |
| Refresh Token 无效 | `REFRESH_TOKEN_INVALID` | 401 |
| 用户被禁用 | `ACCOUNT_DISABLED` | 403 |
| 用户已注销 | `ACCOUNT_DELETED` | 403 |
| 资源不存在 | `RESOURCE_NOT_FOUND` | 404 |
| 邮箱已存在 | `EMAIL_ALREADY_EXISTS` | 409 |
| 标签名称重复 | `TAG_NAME_ALREADY_EXISTS` | 409 |
| 客户端临时 ID 冲突 | `CLIENT_TEMP_ID_CONFLICT` | 409 |
| 服务端异常 | `INTERNAL_ERROR` | 500 |

查询不存在、已删除或不属于当前用户的业务资源时，统一返回 404，避免暴露资源存在性。

### 3.3 数据隔离规则

所有用户业务数据必须通过当前认证用户隔离：

- 查询待办必须附加 `user_id = currentUserId` 和 `deleted_at IS NULL`。
- 查询标签必须附加 `user_id = currentUserId` 和 `deleted_at IS NULL`。
- 待办关联标签前必须校验标签属于当前用户且未删除。
- `todo_tags` 写入时必须同时写入 `user_id`，避免跨用户错误关联。
- 用户注销后，用户信息脱敏，待办和标签软删除，全部令牌撤销。

### 3.4 数据库迁移规则

- 所有表结构变更必须放在 `backend/src/main/resources/db/migration`。
- 命名格式：`V{版本号}__{英文描述}.sql`，例如 `V1__create_initial_schema.sql`。
- 已执行到共享环境的 migration 不允许修改，只能新增版本。
- MVP 的 V1 初始化迁移不创建附件表。

## 4. 阶段 0：工程基础与公共能力

### 4.1 目标

建立后端开发的基础设施，使后续模块能在统一规范下开发、测试和联调。

### 4.2 开发内容

- 补齐 Maven 依赖：MyBatis、Flyway、MySQL Driver、Validation、Security、测试依赖。
- 配置 `application.properties` 或 profile 配置：数据库连接、MyBatis 驼峰映射、Flyway 开关、日志级别。
- 建立公共响应和错误处理：
  - `ApiError`。
  - `ErrorCode`。
  - `GlobalExceptionHandler`。
  - `ValidationErrorDetail`。
- 建立认证上下文：
  - 当前用户 ID 获取能力。
  - 未认证请求拦截。
  - 用户状态校验。
- 建立 UTC 时间处理工具，统一使用 `Instant` 或 `OffsetDateTime`。

### 4.3 交付物

- 后端能启动。
- 全局异常能返回统一 JSON。
- 参数校验错误能返回 `VALIDATION_ERROR`。
- 受保护接口无令牌时返回 401。

### 4.4 验收检查

- 启动应用无异常。
- 添加一个临时健康检查接口可访问。
- 构造非法请求时返回统一错误结构。

## 5. 阶段 1：数据库初始化模块

### 5.1 目标

通过 Flyway 建立 MVP 所需数据库结构，为业务模块提供稳定数据基础。

### 5.2 开发内容

创建 `V1__create_initial_schema.sql`，包含以下表：

- `users`。
- `todos`。
- `tags`。
- `todo_tags`。
- `auth_tokens`。

关键设计必须落实：

- 所有主键使用 `BIGINT UNSIGNED AUTO_INCREMENT`。
- 时间字段使用 `DATETIME(3)`。
- 软删除字段统一为 `deleted_at`。
- `users.normalized_email` 唯一。
- `tags` 使用 `active_key` 支持软删除后同名复用。
- `todos(user_id, client_temp_id)` 唯一，支持离线创建幂等。
- 不创建 `attachments` 表。

### 5.3 交付物

- `backend/src/main/resources/db/migration/V1__create_initial_schema.sql`。
- 与表结构对应的 Entity。
- 与核心表对应的 Mapper 基础 CRUD。

### 5.4 验收检查

- 空库启动时 Flyway 可成功执行。
- 重复启动不会重复创建表。
- 标签软删除后可重新创建同名标签。
- 同一用户相同 `client_temp_id` 不能重复创建待办。

## 6. 阶段 2：认证与用户账户模块

### 6.1 目标

完成注册、登录、刷新令牌、退出、当前用户信息和用户注销，为所有受保护接口提供认证基础。

### 6.2 模块边界

拆为两个模块：

- `user`：用户资料、用户状态、注销脱敏。
- `auth`：登录注册、令牌签发、令牌刷新、令牌撤销。

### 6.3 接口范围

| 方法 | 路径 | 优先级 | 说明 |
| --- | --- | --- | --- |
| POST | `/api/v1/auth/register` | P0 | 注册并返回用户信息和令牌。 |
| POST | `/api/v1/auth/login` | P0 | 登录并返回令牌。 |
| POST | `/api/v1/auth/refresh` | P0 | 刷新 Access Token 和 Refresh Token。 |
| POST | `/api/v1/auth/logout` | P1 | 撤销当前设备令牌。 |
| GET | `/api/v1/me` | P0 | 获取当前用户信息。 |
| DELETE | `/api/v1/me` | P1 | 注销当前用户。 |

### 6.4 核心业务规则

- 邮箱归一化：`trim` 后转小写，写入 `normalized_email`。
- 密码必须哈希存储，禁止明文保存。
- 用户注册后状态为 `ACTIVE`。
- 登录时用户不存在和密码错误返回统一认证失败提示。
- 非 `ACTIVE` 用户禁止登录。
- 服务端只保存令牌哈希，不保存明文令牌。
- Refresh Token 刷新成功后建议轮换，并撤销旧 Refresh Token。
- 退出登录撤销当前设备相关 Access Token 和 Refresh Token。
- 用户注销需要：
  - 校验身份。
  - 用户状态改为 `DELETED`。
  - 脱敏邮箱、昵称、头像。
  - 软删除用户待办和标签。
  - 撤销全部令牌。

### 6.5 开发顺序

1. 用户注册。
2. 密码哈希与邮箱唯一性。
3. 登录与最近登录时间更新。
4. Access Token + Refresh Token 签发。
5. 令牌认证过滤器。
6. Refresh Token 轮换。
7. 当前用户信息。
8. 退出登录。
9. 用户注销。

### 6.6 测试重点

- 新邮箱注册成功。
- 重复邮箱注册返回 `EMAIL_ALREADY_EXISTS`。
- 密码错误登录失败且不暴露用户是否存在。
- Access Token 过期后可用有效 Refresh Token 刷新。
- 已撤销 Refresh Token 不可刷新。
- 注销后原账号不可登录，原令牌不可访问受保护接口。

## 7. 阶段 3：标签模块

### 7.1 目标

完成标签创建、列表、编辑、删除和排序，为待办分类能力提供基础。

### 7.2 接口范围

| 方法 | 路径 | 优先级 | 说明 |
| --- | --- | --- | --- |
| GET | `/api/v1/tags` | P0 | 查询标签列表，包含未完成待办数量。 |
| POST | `/api/v1/tags` | P0 | 创建标签。 |
| PUT | `/api/v1/tags/{id}` | P0 | 更新标签。 |
| DELETE | `/api/v1/tags/{id}` | P0 | 删除标签。 |
| PUT | `/api/v1/tags/reorder` | P1 | 批量更新排序。 |

### 7.3 核心业务规则

- 标签名称必填，长度 1-30。
- 标签颜色为空时默认 `#2F80ED`。
- 标签颜色必须符合 `^#[0-9A-Fa-f]{6}$`。
- 标签名称归一化后，同一用户下未删除标签不可重复。
- 删除标签采用软删除，并将 `active_key` 更新为标签 ID。
- 删除标签不删除待办。
- 标签列表按 `sort_order ASC, created_at ASC` 排序。
- 标签列表返回未完成待办数量，只统计未删除且状态为 `TODO` 的待办。
- 排序更新必须在事务内完成，并校验所有标签属于当前用户。

### 7.4 开发顺序

1. 创建标签。
2. 标签列表。
3. 编辑标签。
4. 删除标签。
5. 标签排序。

### 7.5 测试重点

- 同一用户重复标签名返回 `TAG_NAME_ALREADY_EXISTS`。
- 不同用户可创建相同标签名。
- 删除后可重新创建同名标签。
- 越权编辑、删除其他用户标签返回 404。
- 标签列表的未完成待办数量准确。

## 8. 阶段 4：待办事项模块

### 8.1 目标

完成待办 CRUD、完成状态切换、标签关联、提醒时间保存、分页筛选和离线创建幂等同步。

### 8.2 接口范围

| 方法 | 路径 | 优先级 | 说明 |
| --- | --- | --- | --- |
| GET | `/api/v1/todos` | P0/P1 | 查询待办列表，支持分页、搜索、筛选。 |
| POST | `/api/v1/todos` | P0/P1 | 创建待办，支持 `clientTempId` 幂等。 |
| GET | `/api/v1/todos/{id}` | P0 | 查询待办详情。 |
| PUT | `/api/v1/todos/{id}` | P0 | 更新待办。 |
| PATCH | `/api/v1/todos/{id}/status` | P0 | 完成或取消完成。 |
| DELETE | `/api/v1/todos/{id}` | P0 | 删除待办。 |

### 8.3 核心业务规则

- 标题必填，长度 1-120。
- 描述最多 5000 字符。
- 优先级支持 `LOW`、`MEDIUM`、`HIGH`，默认 `MEDIUM`。
- 新建待办默认状态为 `TODO`。
- 创建成功后 `sync_status` 为 `SYNCED`。
- 创建和更新时，`tagIds` 中所有标签必须属于当前用户且未删除。
- 更新标签关系时采用整体替换。
- 查询详情时，不存在、已删除或不属于当前用户统一返回 404。
- 标记为 `DONE` 时写入 `completed_at`。
- 改回 `TODO` 时清空 `completed_at`。
- 删除待办采用软删除，普通列表和详情不可见。
- 提醒时间只保存，不触发服务端推送。

### 8.4 离线创建幂等规则

当 `clientTempId` 非空时：

1. 先按 `(userId, clientTempId)` 查询未删除待办。
2. 如果存在，直接返回已有待办详情。
3. 如果不存在，创建新待办。
4. 并发重复提交时，捕获唯一索引冲突后再次查询并返回已有待办。

同一个离线待办重复同步不得产生重复数据。

### 8.5 列表查询规则

支持查询参数：

- `status`。
- `priority`。
- `tagId`。
- `keyword`。
- `dueFrom`。
- `dueTo`。
- `sort`。
- `page`。
- `size`。

默认规则：

- 只返回当前用户未删除待办。
- 分页大小最大 100。
- 默认排序：未完成优先，其次截止时间升序，再更新时间降序。
- 关键词搜索范围为标题和描述，MVP 使用 `LIKE`。
- 标签筛选必须校验标签属于当前用户。

### 8.6 开发顺序

1. 创建待办，不含标签。
2. 待办详情。
3. 待办列表基础分页。
4. 标签关联创建。
5. 编辑待办并整体替换标签。
6. 完成和取消完成。
7. 删除待办。
8. 离线创建幂等。
9. 关键词搜索和组合筛选。
10. 提醒时间字段保存与返回。

### 8.7 测试重点

- 标题为空或超长返回 `VALIDATION_ERROR`。
- 使用其他用户标签创建或更新待办失败。
- 用户只能看到自己的待办。
- 删除后列表不可见，详情返回 404。
- 同一 `clientTempId` 重试两次只生成一条待办。
- 状态从 `TODO` 到 `DONE` 时写入完成时间，反向操作时清空。
- 组合筛选结果符合用户、状态、标签、优先级和截止时间条件。

## 9. 阶段 5：搜索、筛选与性能收口

### 9.1 目标

补齐 P1 搜索筛选能力，并保证单用户 10,000 条待办数据内常用查询可接受。

### 9.2 开发内容

- MyBatis XML 使用 `<where>` 和 `<if>` 组织动态条件。
- 关键词搜索使用参数绑定，禁止字符串拼接 SQL。
- 列表查询必须走 `user_id` 和 `deleted_at` 过滤。
- 按标签筛选使用 `todo_tags.user_id` 加强隔离。
- 检查索引是否覆盖常用条件：
  - `idx_todos_user_status_due`。
  - `idx_todos_user_updated`。
  - `idx_todos_user_priority`。
  - `idx_todo_tags_user_tag`。

### 9.3 验收检查

- 常用列表接口 95% 情况下响应时间目标不超过 500 ms。
- 搜索其他用户待办关键词不会返回结果。
- 组合筛选可与分页同时生效。
- `size` 超过 100 时按上限处理或返回校验错误。

## 10. 阶段 6：接口联调与前后端契约

### 10.1 目标

使 Flutter 客户端可以基于稳定契约完成注册、登录、创建待办、标签管理、搜索筛选和离线同步主流程。

### 10.2 联调顺序

1. 注册、登录、刷新令牌。
2. 当前用户信息。
3. 标签创建、列表、编辑、删除。
4. 待办创建、列表、详情。
5. 待办编辑、完成、删除。
6. 待办按标签筛选。
7. 关键词搜索和组合筛选。
8. 退出登录。
9. 离线创建重复同步。
10. 用户注销。

### 10.3 契约检查清单

- 路径统一带 `/api/v1`。
- 错误响应结构一致。
- 分页字段命名一致。
- 时间字段格式一致。
- 空数组返回 `[]`，不返回 `null`。
- 可选字段为空时响应策略一致。
- 401、403、404、409 语义清晰。

## 11. 阶段 7：测试与验收

### 11.1 单元测试

Service 层至少覆盖：

- 注册、登录、刷新令牌。
- 用户注销。
- 标签唯一性和软删除。
- 待办创建、编辑、删除。
- 待办标签归属校验。
- 离线创建幂等。

### 11.2 接口测试

Controller 层至少覆盖：

- 参数校验失败。
- 未认证访问。
- 正常请求响应结构。
- 越权访问返回 404。
- 冲突场景返回 409。

### 11.3 集成测试

Repository 或集成测试至少覆盖：

- Flyway 空库迁移。
- 待办列表组合筛选。
- 标签列表未完成数量。
- Refresh Token 查询和撤销。
- 用户注销事务性更新。

### 11.4 MVP 验收主流程

后端 MVP 必须通过以下闭环：

1. 用户注册成功并获得令牌。
2. 使用令牌创建标签。
3. 创建待办并关联标签。
4. 查询待办列表可见新待办。
5. 按标签筛选可返回该待办。
6. 编辑待办标题、描述、提醒时间和标签。
7. 标记待办完成。
8. 删除待办后列表不可见。
9. 退出登录后原令牌不可访问受保护接口。
10. 使用相同 `clientTempId` 重复创建待办不会生成重复记录。

| 阶段 1 | 数据库初始化 | P0 | V1 Flyway migration、Entity、基础 Mapper。 |
| 阶段 2 | 认证与用户账户 | P0/P1 | 注册、登录、刷新、退出、当前用户、注销。 |
| 阶段 3 | 标签模块 | P0/P1 | 标签 CRUD、列表数量、排序。 |
| 阶段 4 | 待办模块 | P0/P1 | 待办 CRUD、状态、标签关联、离线幂等、提醒字段。 |

## 13. 暂不实现范围

以下能力不得混入 MVP 主线，避免扩大开发范围：

- 图片上传、展示、删除。
- 图片缩略图生成。
- 本地文件或对象存储方案。
- 服务端推送通知。
- 子任务。
- 循环任务。
- 团队协作。
- 独立管理后台。

如需为 P2 预留扩展点，只保留清晰边界，不在 V1 migration 中创建附件表，不暴露附件接口。

## 14. 开发完成定义

每个模块完成时必须满足：

- 接口符合 SRS 和功能需求文档路径、字段、状态码约定。
- Service 层业务规则完整。
- Mapper 查询默认包含用户隔离和软删除过滤。
- 关键异常场景返回统一错误码。
- 关键路径有测试覆盖。
- 不引入 P2 范围功能。
- 可与前端联调，并能进入下一阶段开发。

MVP 完成时必须满足：

- 所有 P0 功能完成。
- P1 中退出、注销、离线创建同步、标签排序、提醒字段、搜索筛选至少达到测试版可用。
- 新用户能完成注册、登录、创建标签、创建待办、筛选待办、完成待办、删除待办和退出登录。
- 离线创建恢复同步不会重复创建待办。
- Flyway 可在空库上完成初始化。
