# 待办事项 App 软件需求规格说明书

版本：1.0  
日期：2026-07-01  
项目代号：BEGO  
技术栈：Flutter + Spring Boot + MyBatis + MySQL + Flyway

## 1. 引言

### 1.1 目的

本文档定义待办事项 App 的业务目标、功能需求、非功能需求、数据需求、接口约定和验收标准，为后续产品设计、后端开发、Flutter 客户端开发、数据库迁移和测试提供统一依据。

### 1.2 范围

本系统面向个人用户，提供待办事项的创建、管理、提醒、标签归类、搜索筛选和基础账户能力。第一阶段以单用户在线同步为核心，图片附件上传延后到后续阶段，不包含团队协作、聊天、复杂项目管理和第三方日历深度集成。

### 1.3 目标用户

- 需要记录日常事项、工作事项、学习任务的个人用户。
- 需要通过标签、优先级、截止时间快速整理任务的轻量效率工具用户。
- 后续需要给任务附加图片作为凭证、参考图或过程记录的用户。

### 1.4 术语

| 术语 | 说明 |
| --- | --- |
| Todo | 待办事项，是系统的核心任务实体。 |
| Tag | 用户自定义标签，用于对待办事项分类。 |
| Attachment | 附件，后续阶段主要指图片附件。 |
| Reminder | 提醒时间，用于在截止前或指定时间提示用户。 |
| Soft Delete | 软删除，数据标记为删除但不立即物理删除。 |
| Flyway Migration | Flyway 数据库迁移脚本，用于维护数据库结构版本。 |

## 2. 总体描述

### 2.1 产品定位

BEGO 待办事项 App 是一款轻量、清晰、可扩展的个人任务管理工具。产品重点是快速记录、稳定同步、灵活标签和清晰筛选，避免复杂项目管理软件的高学习成本。

### 2.2 系统架构

系统采用前后端分离架构：

- 客户端：Flutter，支持 Android、iOS，后续可扩展 Web/Desktop。
- 服务端：Spring Boot，提供 RESTful API。
- 数据库：MySQL，存储用户、待办事项、标签和提醒数据；附件数据后续阶段再引入。
- 数据库迁移：Flyway，所有表结构变化通过版本化 SQL 脚本管理。
- 图片存储：方案未定，上传图片功能延后；后续在本地存储、对象存储或混合方案中确定。

### 2.3 用户角色

| 角色 | 权限 |
| --- | --- |
| 未登录用户 | 注册、登录、查看应用基础信息。 |
| 已登录用户 | 管理自己的待办事项、标签、提醒和个人资料；图片附件管理为后续能力。 |
| 系统管理员 | 第一阶段不提供独立管理后台，可通过数据库或运维工具处理异常数据。 |

客户端：

## 3. 功能需求

### 3.0 优先级定义

| 优先级 | 含义 |
| --- | --- |
| P0 | MVP 必须交付，缺失则产品主流程不可用。 |
| P1 | MVP 建议交付，可在 P0 稳定后继续完善。 |
| P2 | 后续阶段交付，不阻塞 MVP 发布。 |

### 3.1 用户账户

#### FR-001 用户注册（P0）

系统应允许用户使用邮箱和密码注册账户。

验收标准：

- 邮箱必须唯一。
- 密码长度不少于 8 位。
- 注册成功后返回用户基础信息和访问令牌。
- 重复邮箱注册时返回明确错误。

#### FR-002 用户登录（P0）

系统应允许用户使用邮箱和密码登录。

验收标准：

- 登录成功后返回访问令牌。
- 密码错误或用户不存在时返回统一的认证失败提示。
- 访问受保护接口时必须携带有效令牌。

#### FR-003 用户退出（P1）

客户端应支持用户退出登录并清除本地令牌。

验收标准：

- 退出后不能继续访问需要登录的页面。
- 再次打开 App 时保持未登录状态，除非重新登录。
- 服务端应撤销当前设备的 Access Token 和 Refresh Token。

#### FR-004 刷新令牌（P0）

系统应采用 Access Token + Refresh Token 认证方案。

验收标准：

- Access Token 过期后，客户端可使用 Refresh Token 获取新的令牌。
- Refresh Token 失效或过期时，客户端应清除登录态并跳转登录页。
- 服务端不保存明文令牌，只保存令牌哈希。

#### FR-005 用户注销（P1）

用户应能注销自己的账号。

验收标准：

- 注销前需要二次确认。
- 注销后撤销该用户全部令牌。
- 注销后用户 PII 应脱敏，待办和标签等业务数据应软删除。

### 3.2 待办事项管理

#### FR-101 创建待办事项（P0）

用户应能创建待办事项。

字段要求：

- 标题：必填，1-120 个字符。
- 描述：选填，最多 5000 个字符。
- 截止时间：选填。
- 优先级：选填，支持低、中、高。
- 状态：默认未完成。
- 标签：选填，可关联多个用户自定义标签。

验收标准：

- 创建成功后列表立即显示新待办。
- 标题为空时阻止提交。
- 关联的标签必须属于当前用户。

#### FR-107 离线创建待办（P1）

客户端应支持在离线状态下创建待办，并在网络恢复后自动同步。

验收标准：

- 离线创建的待办应生成客户端临时 ID。
- 首页应展示待同步状态。
- 同步成功后使用服务端 ID 替换客户端临时 ID。
- 重复同步同一客户端临时 ID 不应创建重复待办。

#### FR-102 查看待办列表（P0）

用户应能查看自己的待办事项列表。

列表应支持：

- 按创建时间、更新时间、截止时间排序。
- 按状态筛选：全部、未完成、已完成。
- 按优先级筛选。
- 按标签筛选。
- 按关键词搜索标题和描述。
- 分页或游标加载。

验收标准：

- 用户只能看到自己的待办事项。
- 筛选、搜索和排序条件可以组合使用。
- 默认排序为未完成优先，其次按截止时间升序，再按更新时间降序。

#### FR-103 查看待办详情（P0）

用户应能查看单个待办事项的完整内容。

详情应包含：

- 标题、描述、状态、优先级、截止时间。
- 创建时间、更新时间、完成时间。
- 关联标签。
- 提醒信息。

验收标准：

- 访问不存在或不属于当前用户的待办时返回 404。

#### FR-104 编辑待办事项（P0）

用户应能修改待办事项的标题、描述、截止时间、优先级、标签和提醒。

验收标准：

- 修改后更新时间自动更新。
- 删除或替换标签关系时不能影响标签本身。
- 并发更新时以后端最后一次成功提交为准，后续可扩展乐观锁。

#### FR-105 完成和取消完成（P0）

用户应能将待办标记为已完成，也应能取消完成。

验收标准：

- 标记完成时记录完成时间。
- 取消完成时清空完成时间。
- 已完成任务默认不触发未来提醒。

#### FR-106 删除待办事项（P0）

用户应能删除待办事项。

验收标准：

- 删除采用软删除。
- 删除待办时同时隐藏其标签关联关系。
- 被删除的待办不出现在普通列表和搜索结果中。

### 3.3 自定义标签

#### FR-201 创建标签（P0）

用户应能创建自定义标签。

字段要求：

- 名称：必填，1-30 个字符。
- 颜色：选填，使用十六进制颜色值，例如 `#2F80ED`。
- 排序值：选填，用于客户端展示顺序。

验收标准：

- 同一用户下标签名称不可重复。
- 标签颜色为空时使用系统默认色。

#### FR-202 标签列表（P0）

用户应能查看自己的全部标签。

验收标准：

- 返回每个标签关联的未完成待办数量。
- 默认按排序值升序、创建时间升序展示。

#### FR-203 编辑标签（P0）

用户应能修改标签名称、颜色和排序值。

验收标准：

- 修改名称时仍需保证同一用户下唯一。
- 修改标签后，已关联待办的标签展示同步更新。

#### FR-204 删除标签（P0）

用户应能删除标签。

验收标准：

- 删除标签不删除待办事项。
- 删除后待办与该标签的关联解除。
- 删除采用软删除。

### 3.4 图片附件（P2，后续阶段）

图片附件上传功能暂不纳入 MVP。当前阶段只保留产品和数据库扩展设计，不要求实现上传、展示、删除、缩略图生成和存储清理。

#### FR-301 上传图片（P2）

用户应能给待办事项上传图片附件。

验收标准：

- 支持 JPG、JPEG、PNG、WEBP。
- 单张图片大小默认不超过 10 MB。
- 单个待办最多上传 9 张图片。
- 上传成功后返回附件 ID、文件名、大小、MIME 类型、访问 URL 和缩略图 URL。
- 上传接口必须校验图片类型，不仅依赖文件扩展名。

#### FR-302 查看图片（P2）

用户应能在待办详情中查看图片缩略图，并打开原图。

验收标准：

- 用户只能查看自己待办下的附件。
- 图片访问 URL 不应暴露服务端真实文件系统路径。

#### FR-303 删除图片（P2）

用户应能删除待办事项中的某张图片。

验收标准：

- 删除附件采用软删除。
- 删除后详情页不再展示该图片。
- 文件物理清理由后台任务或运维策略处理。

### 3.5 提醒

#### FR-401 设置提醒（P1）

用户应能为待办设置一个提醒时间。

验收标准：

- 提醒时间可以早于或等于截止时间。
- 如果没有截止时间，也可以设置独立提醒时间。
- 已完成或已删除待办不应继续触发提醒。

#### FR-402 本地通知（P1，测试版）

Flutter 客户端应支持基于提醒时间的本地通知。

验收标准：

- 客户端获得通知权限后，可在指定时间提醒用户。
- 用户更新或删除提醒后，本地通知计划同步更新。
- 当前阶段为测试版本地通知，不要求达到正式发布规格。

备注：第一阶段可以不实现服务端推送通知；后续可扩展 APNs/FCM。

### 3.6 搜索与筛选

#### FR-501 关键词搜索（P1）

用户应能通过关键词搜索待办标题和描述。

验收标准：

- 支持模糊匹配。
- 搜索范围仅限当前用户未删除数据。

#### FR-502 组合筛选（P1）

用户应能组合使用状态、标签、优先级、截止时间范围进行筛选。

验收标准：

- 筛选结果分页返回。
- 筛选条件变更后客户端列表刷新。

### 3.7 客户端体验

#### FR-601 首页（P0）

首页应展示待办列表和常用筛选入口。

验收标准：

- 用户可以快速新增待办。
- 用户可以快速切换全部、今天、即将到期、已完成。
- 列表项应展示标题、标签、优先级、截止时间和完成状态。

#### FR-602 新建和编辑页（P0）

新建和编辑页应提供完整任务编辑能力。

验收标准：

- 表单有明确校验提示。
- 网络失败时不丢失当前输入内容。

#### FR-603 标签管理页（P0）

标签管理页应支持创建、编辑、删除和排序标签。

验收标准：

- 标签颜色可视化展示。
- 删除标签前需要二次确认。

### 4.2 索引建议

- `users.email` 唯一索引。
- `todos(user_id, deleted_at, status, due_at)` 普通索引。
- `todos(user_id, deleted_at, updated_at)` 普通索引。
- `tags(user_id, deleted_at, name)` 唯一约束需兼容软删除策略。
- `todo_tags(tag_id, todo_id)` 普通索引。
- `attachments(todo_id, deleted_at)` 普通索引，P2 阶段引入。

### 4.3 Flyway 迁移要求

- 所有数据库结构变更必须通过 `backend/src/main/resources/db/migration` 下的 Flyway 脚本提交。
- 命名格式：`V{版本号}__{英文描述}.sql`，例如 `V1__create_initial_schema.sql`。
- 迁移脚本必须可重复在空库执行成功。
- 已合入主分支的迁移脚本不得修改，只能追加新版本。
- 测试环境和生产环境使用同一套迁移脚本。

## 5. 接口需求

### 5.1 通用约定

基础路径建议：`/api/v1`

认证：

- 受保护接口使用 `Authorization: Bearer <token>`。
- 返回 `401` 表示未认证或令牌失效。
- 返回 `403` 表示认证成功但无权限。

通用错误响应：

```json
{
  "code": "VALIDATION_ERROR",
  "message": "请求参数不合法",
  "details": [
    {
      "field": "title",
      "message": "标题不能为空"
    }
  ]
}
```

分页响应：

```json
{
  "items": [],
  "page": 1,
  "size": 20,
  "total": 100
}
```

### 5.2 认证接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/auth/register` | 注册 |
| POST | `/auth/login` | 登录 |
| POST | `/auth/refresh` | 刷新 Access Token 和 Refresh Token |
| POST | `/auth/logout` | 退出 |
| GET | `/me` | 获取当前用户信息 |
| DELETE | `/me` | 注销当前用户 |

### 5.5 附件接口（P2，后续阶段）

附件接口暂不纳入 MVP，待图片存储方案确定后再实现。

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| POST | `/todos/{todoId}/attachments` | 上传图片 |
| GET | `/attachments/{id}` | 查询附件元数据 |
| DELETE | `/attachments/{id}` | 删除附件 |

上传接口要求：

- 请求类型：`multipart/form-data`。
- 文件字段名：`file`。
- 服务端应返回附件元数据。

## 6. 非功能需求

### 6.1 性能

- 待办列表首屏接口在 95% 情况下响应时间不超过 500 ms。
- P2 图片上传接口在网络正常情况下应提供明确上传状态。
- 单用户 10,000 条待办数据内，常用筛选查询应保持可接受响应速度。

### 6.2 安全

- 密码必须使用安全哈希算法存储，不能明文保存。
- 所有用户数据接口必须基于当前认证用户做数据隔离。
- P2 上传文件必须校验 MIME 类型和文件内容。
- P2 图片 URL 不应泄漏服务器本地路径。
- 错误信息不应暴露堆栈、SQL、文件系统路径等敏感信息。

### 6.3 可用性

- 客户端关键操作应有加载、成功、失败状态。
- 网络异常时应给出可理解提示。
- 删除待办、删除标签等破坏性操作需要二次确认；P2 图片删除也应二次确认。

### 6.4 可维护性

- 后端按 Controller、Service、Repository、DTO、Domain 分层。
- 数据库结构变更统一通过 Flyway。
- API 返回结构保持稳定，破坏性变更需升级版本路径。
- 客户端状态管理方案应统一，不在页面中散落复杂业务逻辑。

### 6.5 兼容性

- 后端使用 MySQL 8.x 作为目标数据库。
- 时间字段统一使用 UTC 存储，客户端按本地时区展示。
- API 时间格式使用 ISO 8601。

## 7. 业务规则

- 用户只能操作自己的待办和标签；P2 阶段扩展为附件也必须做用户隔离。
- 一个待办可以有多个标签，一个标签可以关联多个待办。
- 标签删除不影响待办本身。
- 待办删除后，其标签关联在业务上不可见；P2 阶段附件也应随待办隐藏。
- 已完成待办可以重新变为未完成。
- 已完成待办默认不纳入“今天”和“即将到期”的未完成统计。
- P2 阶段附件数量超过限制时，服务端应拒绝新的上传请求。

## 8. 验收标准

### 8.1 MVP 验收范围

MVP 应至少完成：

- 用户注册、登录、退出。
- Access Token + Refresh Token。
- 待办创建、列表、详情、编辑、完成、删除。
- 离线创建待办和恢复网络后的同步。
- 标签创建、列表、编辑、删除。
- 待办关联多个标签。
- 关键词搜索和基础筛选。
- Flyway 初始化表结构脚本。
- Flutter 基础页面流程。

不纳入 MVP：

- 图片上传、展示、删除。
- 图片缩略图生成、对象存储、本地文件清理。

### 8.2 测试要求

后端：

- Service 层核心业务单元测试。
- Controller 层接口测试。
- Repository 或集成测试覆盖主要查询条件。
- Flyway migration 在空库上可执行。

客户端：

- 表单校验测试。
- 列表展示和筛选交互测试。
- P2 阶段补充上传图片失败和成功状态测试。

### 8.3 退出准则

- MVP 范围内所有 P0/P1 缺陷关闭。
- 主要接口有文档或测试覆盖。
- 新用户可以从注册开始完整完成创建待办、添加标签、筛选待办、完成待办的闭环。
- 离线创建的待办在网络恢复后可以同步到服务端且不会重复创建。

## 10. 风险与约束

- 图片存储方案尚未确定，图片上传功能延后；后续如果从本地文件切换到对象存储，需要提前抽象存储服务接口。
- 本地通知在不同移动平台上的权限和后台策略存在差异，需要客户端分别适配。
- 软删除和唯一约束需要在 MySQL 中谨慎设计，避免删除后无法重新创建同名标签。
- 搜索初期可使用 `LIKE`，数据量增长后可评估 MySQL Full-Text 或专用搜索服务。
