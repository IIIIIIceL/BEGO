# 考研与日常待办手机应用设计方案-初版

## 1. 产品定位

本应用定位为一个面向考研和日常生活的智能待办与复盘工具，帮助用户记录任务、安排时间、追踪学习进度，并通过 Agent 生成总结、提醒、建议和计划。

它不只是普通 Todo List，而是偏向：

- 长期目标管理
- 每日任务执行
- 考研学习记录
- 数据统计复盘
- Agent 智能辅助

核心目标是帮助用户形成稳定的学习和生活管理闭环。

## 2. 核心使用场景

### 2.1 考研学习记录

用于记录和管理考研相关任务，例如：

- 单词背诵
- 数学刷题
- 政治复习
- 专业课阅读
- 模拟卷练习
- 错题复盘
- 每日学习时长统计
- 每周、每月复盘

### 2.2 日常生活待办

用于记录日常生活事项，例如：

- 买东西
- 交费
- 取快递
- 锻炼
- 预约事项
- 整理资料
- 临时提醒

### 2.3 长期目标拆解

用于将长期目标拆解为阶段任务和每日任务，例如：

- 90 天完成数学一轮复习
- 30 天背完核心词汇
- 每周完成一套英语真题
- 每月完成一次综合复盘

### 2.4 Agent 辅助

Agent 用于处理总结、分析、规划等事务，例如：

- 生成每日总结
- 生成每周复盘
- 分析未完成任务原因
- 根据目标生成明日计划
- 识别任务过载
- 回答学习进度相关问题

## 3. 功能模块设计

## 3.1 待办事项模块

待办事项模块是应用的基础能力。

每个任务建议包含以下字段：

| 字段 | 说明 |
| --- | --- |
| 标题 | 任务名称 |
| 描述 | 任务详细说明 |
| 分类 | 学习、生活、工作等 |
| 优先级 | 低、中、高 |
| 截止时间 | 任务最晚完成时间 |
| 提醒时间 | 通知提醒时间 |
| 重复规则 | 每天、每周、每月、自定义 |
| 预计耗时 | 用户预估任务时长 |
| 实际耗时 | 实际完成用时 |
| 完成状态 | 未开始、进行中、已完成、已延期 |
| 关联目标 | 绑定长期目标 |
| 标签 | 自定义标签 |
| 备注 | 补充说明 |
| 创建时间 | 任务创建时间 |
| 完成时间 | 任务完成时间 |

任务类型建议包括：

- 普通任务
- 学习任务
- 习惯任务
- 长期目标任务
- 临时记录

基础功能包括：

- 新建任务
- 编辑任务
- 删除任务
- 标记完成
- 延期任务
- 设置提醒
- 设置重复任务
- 设置优先级
- 按日期、分类、标签、目标筛选
- 搜索任务

## 3.2 考研专区

考研专区是本应用区别于普通待办应用的关键模块。

建议单独设计一个“考研”页面，包含：

- 今日学习计划
- 科目进度
- 学习时长
- 错题或问题记录
- 每日复盘
- 每周复盘
- 考研倒计时
- 目标完成率

科目可以由用户自定义，默认可提供：

- 英语
- 数学
- 政治
- 专业课
- 复试
- 其他

学习任务建议额外绑定以下字段：

| 字段 | 说明 |
| --- | --- |
| 科目 | 所属考研科目 |
| 资料名称 | 教材、课程、题集名称 |
| 章节 | 学习章节 |
| 页码范围 | 阅读或练习范围 |
| 题目数量 | 刷题数量 |
| 学习类型 | 看课、刷题、背诵、复盘、模拟 |
| 完成质量评分 | 1 到 5 分 |
| 备注 | 学习感受或问题 |

示例：

```text
任务：高数强化第 3 讲
科目：数学
类型：看课 + 刷题
预计时间：120 分钟
完成质量：4/5
备注：极限部分还不熟，需要二刷
```

## 3.3 日历与时间线

应用建议提供两种主要视图：

- 今日视图
- 日历视图

今日视图用于执行任务，可以按时间段展示：

- 上午
- 下午
- 晚上
- 已完成
- 未完成
- 已逾期

日历视图用于回顾和复盘，可以展示：

- 每天完成了什么
- 每天学习了多久
- 哪些任务延期
- 哪些科目投入较少
- 连续学习天数

## 3.4 统计分析模块

统计分析模块对考研用户非常重要，建议统计：

- 每日完成任务数
- 每日学习时长
- 每周学习时长
- 各科目学习占比
- 任务完成率
- 延期任务数
- 连续学习天数
- 长期目标进度
- 高频未完成任务

可视化形式建议包括：

- 折线图：学习时长趋势
- 柱状图：每日任务完成数
- 饼图：各科目时间占比
- 进度条：长期目标进度
- 热力图：每日学习活跃度

## 6. 技术架构建议

你计划部署云服务器，整体架构可以设计为：

```text
Flutter App
   |
   | HTTPS API
   |
Spring Boot Backend
   |
   | 数据库
   |
MySQL / PostgreSQL
   |
Agent Service
   |
LLM API / 第三方模型
```

### 6.1 前端技术选型

前端统一使用 Flutter 框架。当前项目本身就是一个 Flutter 前端项目，因此不再引入 Kotlin、Jetpack Compose 或其他前端框架作为主路线。

Flutter 端建议采用：

- Flutter
- Dart
- Material Design 3
- Riverpod / Bloc 状态管理
- Dio 网络请求
- Drift / sqflite 本地数据库
- shared_preferences / flutter_secure_storage 保存本地设置和敏感配置
- flutter_local_notifications 实现本地提醒

第一阶段可以先以 Android 为主要目标平台，保留 Flutter 后续扩展到 iOS、Web 或桌面端的可能性。

### 6.2 后端技术选型

后端推荐选择 Spring Boot。

主要原因：

- Spring Boot 生态成熟，认证、数据库、定时任务、接口文档、部署方案都比较完善。
- Java 后端资料多，遇到问题更容易排查。
- 对于任务管理、学习记录、统计分析、Agent 调用这类业务，Spring Boot 的工程化能力比较适合长期维护。
- 未来如果功能扩展到多用户、同步、权限控制、后台任务，Spring Boot 的结构会更稳。

考虑到服务器预算有限，初期大概率使用 2 核 4G 云服务器，后端设计需要控制资源占用。

推荐后端组合：

- Spring Boot
- Spring Web
- Spring Validation
- Spring Security + JWT
- MyBatis-Plus / Spring Data JPA
- MySQL 或 PostgreSQL
- Flyway / Liquibase 数据库迁移
- Docker 部署
- Nginx 反向代理

Redis 可以暂时不作为 MVP 必选项。早期用户量小、功能简单时，登录状态使用 JWT，普通缓存可以先依赖数据库和应用内逻辑。等出现高频统计、消息队列、分布式任务或限流需求时，再引入 Redis。

Agent 能力建议先放在 Spring Boot 后端中作为一个独立模块实现，不必一开始拆成微服务。这样部署简单，资源占用低，也方便快速迭代。后续如果 Agent 调用变复杂，再拆分为独立服务。

核心后端模块：

- 用户模块
- 任务模块
- 分类模块
- 标签模块
- 学习记录模块
- 统计模块
- Agent 模块
- 同步模块
- 通知模块

## 8. Agent 数据流设计

Agent 不建议直接操作全部数据库，而是通过后端提供的受控接口读取和写入数据。

以“帮我总结今天”为例：

```text
用户请求：帮我总结今天
   |
后端查询：
- 今日完成任务
- 今日未完成任务
- 今日学习记录
- 各科目学习时长
- 昨日遗留任务
   |
组装 Prompt
   |
调用 LLM
   |
返回总结
   |
保存 agent_summaries
```

Agent 输出建议采用结构化格式：

```json
{
  "summary": "今天整体完成情况不错...",
  "problems": ["政治任务延期", "数学耗时超预期"],
  "suggestions": ["明天优先处理政治", "数学复盘错题"],
  "tomorrowTasks": [
    {
      "title": "政治选择题复盘",
      "estimatedMinutes": 60
    }
  ]
}
```

这样后续可以支持用户一键接受 Agent 生成的任务。

## 9. MVP 版本范围

第一版建议只做核心闭环：

- 登录 / 注册
- 任务增删改查
- 分类 / 标签
- 今日任务
- 提醒
- 完成记录
- 考研科目管理
- 学习时长记录
- 云同步

第二版再加入：

- 统计图表
- 长期目标
- 每周总结
- 计划自动调整
- 资料进度
- 复盘系统

第三版再加入：

- 对话式 Agent
- 智能拆解目标
- 日程推荐
- 多设备同步
- 导出数据
- 桌面小组件

## 10. 推荐开发顺序

建议按以下顺序推进：

1. 确定数据模型
2. 设计 UI 原型
3. 实现安卓本地任务系统
4. 加入本地提醒
5. 实现后端 API
6. 实现登录和云同步
7. 加入考研学习记录
8. 加入统计模块
9. 接入 Agent 总结
10. 优化体验和数据备份

## 11. 总结

整体建议是先做一个稳定的 Todo + 考研记录工具，再逐步加入 Agent。

第一阶段不要把 Agent 做成全能助手，而是先从“每日总结”和“明日建议”开始。这样技术难度可控，用户价值也明显，后续再扩展到计划调整和对话式智能助手。
